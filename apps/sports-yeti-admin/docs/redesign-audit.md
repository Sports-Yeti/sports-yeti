# SportsYeti Admin — Redesign Audit & Plan

> Generated by `@ux-researcher` + `@ux-architect` + `@brand-guardian` + `@ui-designer` + `@accessibility-auditor` working as one mission. Mirrors the process used to redesign the mobile app in `apps/sports-yeti/docs/design-system.md`.

---

## 1. Scope

The current admin is an Expo / React Native (web target) app at `apps/sports-yeti-admin/`. **28 screens across 17 sidebar items** built directly against the live `services/api.ts` with React Query, no shared UI primitives, and a parallel design palette in `constants/index.ts` (different blue / different spacing / different fonts vs. the mobile app).

This document covers:
1. JTBD framing for the admin user.
2. Heuristic + accessibility audit, severity-ranked.
3. Information architecture redesign.
4. Component / token plan.
5. Execution order for the rebuild.

The follow-up commits implement the plan with mock data only (real API wiring is deliberately deferred).

---

## 2. Primary user — JTBD

The admin user is a **league operator** (often also a referee assignor and venue manager). They wear several hats during a typical season:

| Job | When | Hire criteria |
|---|---|---|
| **Run the day** | Morning open + before each match window | "I want a single Dashboard that shows what needs my attention right now (pending approvals, payment issues, missing waivers, upcoming kickoff times)." |
| **Run competition** | Pre-season + weekly | "I want to design a league season — fixtures, divisions, rosters — and adjust quickly when teams drop or weather hits." |
| **Run venues** | Daily | "I want a calendar view of facility bookings I can drag, confirm, or refund without leaving the page." |
| **Manage people** | Throughout the season | "I want to find any player / referee / parent quickly, see what they owe me and what I owe them, and act in one click." |
| **Move money** | Weekly | "I want to see what came in, what's pending, and refund a payment without opening Stripe directly." |
| **Prove I'm running it well** | End of season + ad-hoc | "I want a report I can hand to my board that explains revenue, participation, and incidents." |

Notably the admin's day is **task-driven**, not document-driven. The current product is the opposite — every page is a CRUD list waiting for a click.

## 3. Heuristic & accessibility audit

Severity scale: **Critical** (blocks task or breaks trust) · **High** (significant friction) · **Medium** (polish/clarity) · **Low** (nice-to-have).

### 3.1 Cross-cutting findings

| ID | Heuristic | Severity | Issue | Fix |
|---|---|---|---|---|
| C-1 | #4 Consistency / brand | Critical | The admin uses an entirely different design language than the mobile product (Material `#3B82F6` blue / inconsistent spacing scale / system fonts). Anyone who jumps between the two products sees two different brands. | Adopt the mobile theme tokens (brand `#006495`, Plus Jakarta + Be Vietnam Pro typography, 8-pt spacing) and re-skin the admin in the same family — but with a denser desktop spacing scale and a darker neutral surface ladder. |
| C-2 | #9 Recover from errors | Critical | Destructive operations (delete league, refund) call `confirm(...)` (web-only browser dialog). On RN native the call throws / is undefined; on web it's an unstyled OS prompt with no context. Errors from API surface as a single inline `Failed to load X` with a generic Retry. | Replace every `confirm()` with a typed `<Modal variant="destructive">` that explains blast radius and requires typed confirmation for irreversible ops (e.g. type the league name). Replace generic error banners with `<Toast>` + inline `<EmptyState>` retry copy that names the actual failure. |
| C-3 | #8 Aesthetic / brand | Critical | Every nav item, dashboard tile, and empty state uses emoji (`📊🏆👥🏟️🏁⛺💳📝📰🏪📈📉💰📋⚙️`). Looks unprofessional in a B2B context next to peer admin tools (Stripe, Linear, Notion, Vercel). | Replace with `lucide-react-native` icons wrapped in `<IconBadge>`. |
| C-4 | #7 Efficiency | Critical | Power users (the entire admin user base) have **no global search and no command palette**. Finding a specific player requires sidebar → People → Players → search → filter. The same applies to teams, leagues, payments, bookings. | Add ⌘K / Ctrl-K command palette with cross-entity search (players, teams, leagues, facilities, payments, bookings, settings). Mock data for now; wire to real search later. |
| C-5 | #1 System status | High | Initial loads are full-page `<ActivityIndicator size="large">` blocking the entire content area. Inline mutations (delete, save) have no per-row loading or saving state. | Replace full-page spinners with `<Skeleton>` rows that match the final table layout. Inline actions get button `loading` state + a `<Toast>` confirmation. |
| C-6 | #2 / i18n | High | Currency is `${Number(value).toFixed(0)}`. No `Intl.NumberFormat`. Dates render as raw ISO strings in lists. Sidebar label says "Divisions" while the API model says `League` — confusing for new admins. | Centralize `formatCurrency`, `formatDate`, `formatRelative`, `formatRange` in `lib/format.ts`. Make label = entity (League). |
| C-7 | a11y / WCAG | High | Every interactive element is bare `TouchableOpacity` with no `accessibilityRole`/`Label`/`Hint`. No focus rings on web. Sidebar has no semantic landmark. Color contrast on `textMuted` (`#94A3B8`) over `surface` (`#FFFFFF`) is **2.86:1** — fails WCAG AA. | Adopt new primitives that ship with `accessibilityRole="button"` baseline + `:focus-visible` styles via `Pressable`. Patch `text.muted` to `#6B7785` (4.6:1). Wrap sidebar in `accessibilityRole="navigation"`. |
| C-8 | #6 Recognition | High | No breadcrumbs. Drilling into League → Team → Player loses context — the title bar always shows the current screen name only. | Add `<Breadcrumbs>` primitive in the page header that surfaces the actual path (Leagues > Aurora Hockey > Glacier Knights > Björn K.). |
| C-9 | #3 User control | High | Forms (LeagueForm, FacilityForm, etc.) have no "discard changes?" guard. Switching nav while editing silently drops state. | Track `dirty` state per form; guard navigation with a `<Modal variant="destructive">` confirm. |
| C-10 | #5 Prevention | High | No bulk operations on lists. Approving 50 pending team applications today requires 50 nav cycles. | Add row selection + bulk-action bar (approve / reject / archive / export). |
| C-11 | #1 Visibility | Medium | Sidebar collapses are not exposed; pinning, customizing visible items, or hiding seasonal areas (Camps off-season) is impossible. | Add a collapse / pin / hide preference per nav group, persisted in `localStorage`. |
| C-12 | #10 Help | Medium | No empty-state hand-holding. A brand-new admin lands on Dashboard with no onboarding. | Add a "Get started" panel for new orgs that shrinks once each step is done (create your first league, add a facility, invite a referee, run a payment test). |
| C-13 | #1 Status / freshness | Medium | Default React Query `staleTime: 5 min` and `refetchOnWindowFocus: true` is reasonable, but there's no visible "Last updated ago" anywhere. | Add a "Updated 2 min ago · Refresh" affordance on the page header for every list / dashboard. |

### 3.2 Per-area findings

#### Dashboard

| ID | Severity | Issue | Fix |
|---|---|---|---|
| D-1 | Critical | The Dashboard fires **8 parallel `useQuery` calls** on mount with full-list `per_page: 100` requests just to compute counts (facilities + waivers). On a real org with thousands of waivers this melts the API. | Compute counts server-side via `getDashboardStats`; remove the per-resource list calls. |
| D-2 | High | Tiles all link to lists, never to actions. The "Pending Team Applications" tile takes you to the full Teams list instead of just the pending queue. | Each tile should deep-link to a *filtered* view (e.g. Teams?status=pending). |
| D-3 | High | "Quick Actions" is dashed-border placeholder grid. The four current actions are arbitrary — no "Refund a payment" or "Resolve a dispute" which are the JTBD-truthy admin actions. | Replace with the top-5 friction-relieving actions: New game, Record payment, Approve waivers, Send announcement, Open booking calendar. |
| D-4 | Medium | "Recent Activity" hard-codes 3 lines based on counts ("X leagues active across the platform"). It's not really activity. | Replace with the most-recent N audit-log rows (already in the system) so admins see real activity. |

#### Leagues

| ID | Severity | Issue | Fix |
|---|---|---|---|
| L-1 | Critical | `Delete league` calls `confirm(...)`. On native this throws. | `<Modal variant="destructive">` that requires typing the league name. |
| L-2 | High | Cards are visually noisy and waste space — fee, location, teams, players are tile stats but the fact that the league is **published vs. draft** doesn't have a visual signal. | Switch list view to a dense table by default with sortable columns. Expose a Table / Cards toggle. Surface `status` (draft / published) prominently. |
| L-3 | High | No bulk actions. Closing a season's leagues at year-end requires deleting one at a time. | Add row selection + bulk archive/duplicate. |

#### Teams

| ID | Severity | Issue | Fix |
|---|---|---|---|
| T-1 | High | Pending team applications surface only via filter — no inbox concept. | Promote pending applications into a dedicated `Approvals` tab with one-click approve / reject. |
| T-2 | High | The captain is shown by avatar only, with no quick way to message them when a team needs attention. | Add a "Message captain" action on the row + Send-DM modal. |

#### Schedule / Games

| ID | Severity | Issue | Fix |
|---|---|---|---|
| S-1 | Critical | The schedule is a list, not a calendar. Spotting weekend-conflicts across leagues is impossible. | Replace with a real week / month calendar view (drag-to-reschedule); list view stays available behind a toggle. |
| S-2 | High | No fixture generator. Today admins manually create every match. | Add a "Generate fixtures" wizard for round-robin / playoff brackets per league. |

#### Facilities + Bookings

| ID | Severity | Issue | Fix |
|---|---|---|---|
| F-1 | Critical | Bookings are managed via separate `BookingDetailModal` / `CreateBookingModal` instead of a calendar. Avoiding double-bookings is manual. | Build a unified Bookings calendar (week + day view) per facility/space, with conflict highlighting. |
| F-2 | High | Refunds are buried inside the booking detail modal and use the legacy `RefundModal`. Most refunds happen from the Payments list. | Move refund flow into `PaymentDetail` (single source of truth) with a confirmation Modal. |

#### People (Players, Referees, Waivers, Camps)

| ID | Severity | Issue | Fix |
|---|---|---|---|
| P-1 | High | Three separate list pages — Players / Referees / Waivers — with no shared search. Finding "everyone Sarah" is three queries. | Unify behind a `People` tab with a `Type` filter (Player / Referee / Coach / Parent). |
| P-2 | High | Waiver completion is opaque — admin sees only signature counts. | Add per-player waiver status grid (signed / required / expired) so the admin can chase the unsigned. |

#### Payments / Finance

| ID | Severity | Issue | Fix |
|---|---|---|---|
| M-1 | Critical | `RefundModal` exists but never explains net impact (tax, processor fee, platform fee). | Refund Modal must show subtotal, processor fees forfeited, net refund, and dispute window. |
| M-2 | High | No reconciliation export. Admins want monthly CSV/PDF of cleared payments. | Export-as-CSV + email-report buttons in `Finance`. |
| M-3 | High | Dashboard "Revenue" tile is gross, not net. Misleading. | Show gross + net + outstanding side-by-side. |

#### Audit / Analytics / Marketplace / News / Stats / Settings

| ID | Severity | Issue | Fix |
|---|---|---|---|
| I-1 | High | These six are scattered top-level nav items used <5% of the time. | Group under an "Insights" sub-nav (Analytics, Stats, Audit, Marketplace, News) and Settings standalone. |
| I-2 | Medium | Audit log has no entity-scoped view from inside (e.g. show me the audit trail for *this league*). | Add an "Activity" tab to every entity detail page that filters audit logs by subject. |

#### Auth (LoginScreen)

| ID | Severity | Issue | Fix |
|---|---|---|---|
| A-1 | High | Error rendered inside a static `<View>` below the form — not announced to screen readers. | Use `<Toast variant="error">` with `accessibilityLiveRegion="assertive"`. |
| A-2 | High | No SSO entry point ("Continue with Google Workspace" / SAML / Magic link). Most B2B SaaS now requires SSO for the admin tier. | Add SSO button shells (mock-only in this pass). |
| A-3 | Medium | "Admin access only" footnote is the only branding. No "Forgot password?" link, no two-factor flow. | Add Forgot password + 2FA shells. |
| A-4 | Medium | No org-picker after login (admins often manage multiple orgs). | Add a post-login Org Switcher. |

## 4. Information Architecture redesign

### 4.1 Sidebar groups (was: 17 flat items, now: 6 groups + Settings)

```
Dashboard
Competition
  Leagues
  Teams
  Schedule
People
  Players
  Referees
  Camps
  Waivers
Venues
  Facilities
  Bookings
Money
  Payments
  Finance
Insights
  Analytics
  Stats
  Audit
  Marketplace
  News
Settings
```

Rationale:
- Reduces cognitive load (Hick's law: 17 → 6 visible groups).
- Maps to the JTBD jobs verbatim.
- Surfaces seasonally-quiet groups (Camps, Insights) under one expandable section instead of cluttering the rail.
- Settings stays at the bottom as a muted target.

### 4.2 Page anatomy (every screen)

```
+-------------------------------------------------------------------+
| Sidebar (260, collapsible to 68)                                  |
|                                                                   |
+-------------------+-----------------------------------------------+
|                   | TopBar: search (cmd-k), notifications, user   |
|                   +-----------------------------------------------+
|                   | Page Header                                   |
|                   |   Breadcrumbs > Page Title                    |
|                   |   Subtitle / metadata (Updated Xm ago)        |
|                   |   Trailing actions (Create, Filters, Export)  |
|                   +-----------------------------------------------+
|                   | Content                                       |
|                   |   - Tabs                                      |
|                   |   - List / Table / Calendar / Form / Detail   |
|                   +-----------------------------------------------+
|                   | Footer / pagination / bulk-actions bar        |
+-------------------+-----------------------------------------------+
```

### 4.3 Page templates

We need 6 page templates, not 28 bespoke pages:
1. **Overview** — dashboard hero with stat cards + alerts + recent activity (Dashboard, Finance, Analytics).
2. **List** — filterable table/cards with bulk actions + pagination (Leagues, Teams, Players, Referees, Facilities, Payments, Waivers, Camps, Audit, News, Marketplace).
3. **Detail** — entity hero + tabs (Overview / Activity / Settings) (LeagueDetail, TeamDetail, PlayerDetail, FacilityDetail, PaymentDetail, RefereeDetail, WaiverDetail, CampDetail).
4. **Form** — multi-step wizard or single-form with autosave (LeagueForm, FacilityForm, WaiverForm, CampForm).
5. **Calendar** — week / month grid with drag (Schedule, Bookings).
6. **Settings** — sidebar + sectioned forms (Settings).

## 8. "Glacier Ethos" alignment pass (Phase 13)

> Generated by `@brand-guardian` + `@ui-designer` + `@ux-architect` after consuming the Stitch reference downloads at `~/Downloads/stitch_glacier_community_prd/` (DESIGN.md + 7 screen exports). Closes the visual-cohesion gap between the admin and the mobile app while leaving every Phase-1–12 feature behavior intact.

### 8.1 Why this pass

Phases 1–12 produced a functionally complete admin (35 screens, mock-only), but visual-language consistency with the mobile product was thin. The audit doc's Section 3 already flagged C-1 ("Two parallel design systems") as Critical. Phases 1–12 partially addressed it via shared brand tokens — but the **chrome itself** (sidebar, top bar, page header, stat cards) still read as a "B2B admin console" rather than the "Alpine Sanctuary" the Glacier ethos calls for.

The Stitch reference made the gap concrete: light atmospheric sidebars, oversized "Summit" hero titles, glassmorphism on floating nav, borderless cards with decorative blobs, and tonal layering instead of 1px lines.

### 8.2 Token additions

| Token | Purpose | Where |
|---|---|---|
| `colors.surface.containerLow / containerHigh / containerLowest` | Tonal layering ladder — replaces 1px borders for section grouping. | `theme/colors.ts` |
| `colors.surface.glassOverlay`, `glassCard` | Frosted overlays at ~78% / 62% opacity for floating nav and glass cards. | `theme/colors.ts` |
| `colors.surface.sidebar / sidebarHover / sidebarActive` | Now light glacier surface (`#F1F5FA`) + brand-soft active row. | `theme/colors.ts` |
| `colors.text.sidebarPrimary / sidebarMuted` | Inverted to read on the new light sidebar (was `#F8FAFC` on dark navy). | `theme/colors.ts` |
| `colors.brand.alpine / alpineSoft / alpineGlow` | Alpine Orange tertiary family for urgent CTAs and high-action statuses (Glacier ethos §2). | `theme/colors.ts` |
| `colors.gradient.alpine` + `colors.gradient.summit` | Alpine sunset gradient + soft glacier wash for hero backdrops. | `theme/colors.ts` |
| `typography.displayLg` (56 / 60 / -1.12) | "Summit" hero title for landing pages. | `theme/typography.ts` |
| `radii.cardLg` (24) and `radii.summit` (32) | Friendly aggressive rounding for hero panels. | `theme/radii.ts` |
| `shadows.glow` | Ambient `on-surface` 6% / 32px blur — alpine elevation, not a hard drop shadow. | `theme/shadows.ts` |
