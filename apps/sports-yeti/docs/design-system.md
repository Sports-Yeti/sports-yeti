### 1.4 Gap analysis

**Missing primitives** (needed by both modern + legacy screens):

| Primitive | Why needed | Used in lieu of (today) |
|---|---|---|
| `Input` (text field with label/help/error) | Login, Register, CreateGame, HighlightUpload, ChatScreen, Search | Raw `TextInput` everywhere |
| `Toast` (non-blocking feedback) | Join game success, payment confirmation, errors | `Alert.alert` in 16 screens |
| `Modal` (centered dialog) | Confirmations, picker-style selectors | `Alert.alert` |
| `BottomSheet` | Filter pickers, sort, action menus, share | None (filters do nothing today) |
| `Tabs` (segmented control) | Marketplace (`Open Games / Slots / Sub Requests`), Bookings filters, referee dashboard | Custom underline-style buttons |
| `Tag` (small status chip) | Live/Booked/Confirmed/Pending/Cancelled badges, sport types | `View` + `Text` ad-hoc |
| `ProgressBar` | Form completion, registration capacity (`12/24 spots`), waiver progress | None |
| `Skeleton` | Replace `ActivityIndicator` blocks during initial load (Dashboard, Marketplace, Messages) | `<ActivityIndicator size="large" />` full-screen |
| `EmptyState` | "No games", "No bookings", "No camps" — repeated 12+ times | Inline emoji + `<Text>` |

**Missing UX patterns:**

- No global error boundary visible to users (errors land in `Alert.alert` or silent `console.error`)
- No real loading-skeleton; full-screen spinners block content
- No pull-to-refresh affordance shown anywhere outside `RefreshControl` (no first-time hint)
- No "you're offline" banner / network state surface
- Filter chips on Discover/Squads do not actually filter (`onPress={() => undefined}`)
- No haptics on key actions (join game, mark all read, scan success)
- No screen-transition motion design (default native push only)

## Step 2 — Heuristic & flow audit

Severity scale: **🔴 Critical** (blocks task or violates trust) · **🟠 High** (significant friction) · **🟡 Medium** (polish/clarity) · **🟢 Low** (nice-to-have).

### 2.1 Cross-cutting findings (apply to most/all screens)

| ID | Heuristic | Severity | Issue | Fix | Cite |
|---|---|---|---|---|---|
| C-1 | **Consistency & standards** (Nielsen #4) | 🔴 | Two parallel design systems coexist. The same "Sports Yeti" wordmark uses `#006495` (modern) on the Discover header and `#1E88E5` (legacy) on Login. Users who Sign In land on a different visual language than every other screen. | Migrate every legacy screen to `theme` + `ui/` primitives. Delete `constants/COLORS|SPACING|FONT_SIZES`. Track via a typed lint rule (no import of the deprecated symbols). | `constants/index.ts:8-44` vs `theme/colors.ts` |
| C-2 | **Help users recognize, diagnose & recover from errors** (Nielsen #9) | 🔴 | 16 screens use `Alert.alert` for both errors and successes (41 instances). Modal alerts block the entire UI, can't be undone, and have no copy-able details. The "Login Failed" alert reads stale `error` from the previous render (race condition: `Alert.alert('Login Failed', error || 'Invalid credentials')` runs synchronously after `await login(...)` rejects, before the next render flushes the new `error`). | Replace all `Alert.alert` calls with `<Toast>` (success/info) and inline `<Input.Error>` (form errors). For destructive confirmations, use the new `<Modal>` with `variant="destructive"`. | `screens/auth/LoginScreen.tsx:36`, `screens/marketplace/MarketplaceScreen.tsx:92`, +14 more |
| C-3 | **Accessibility — perceivable** (WCAG 1.4) | 🔴 | Only **3 of 37** screen files have any `accessibilityLabel/Hint/Role` props. Pressables across the legacy screens are bare `TouchableOpacity` with text children — VoiceOver will announce only the visible label, with no role hint. Filter chips, "see all", emoji-icons, and avatar-as-greeting buttons are unlabeled. | Adopt the new primitives (Button/Chip/IconBadge already set `accessibilityRole`). For raw `Pressable` elsewhere, require `accessibilityRole="button"` + `accessibilityLabel`. Add eslint rule `react-native/no-raw-text` and `eslint-plugin-react-native-a11y`. | All `screens/*` legacy files |
| C-4 | **Visibility of system status** (Nielsen #1) | 🟠 | Initial loads show full-screen `ActivityIndicator size="large"`. Users see a spinner-then-pop transition with no preview of the layout. The Marketplace & Dashboard "shimmer in" entire screens at once. | Use `<Skeleton>` matching the final layout (card outlines + avatar circles). Limit `ActivityIndicator` to inline action confirmations (e.g., the `joinButton` already does this — keep that pattern). | `screens/home/DashboardScreen.tsx:58-64`, `screens/marketplace/MarketplaceScreen.tsx:231-237` |
| C-5 | **Match between system & real world** (Nielsen #2) | 🟡 | Emoji icons (`🏀⚽🏈🏐⚾`) on `CreateGameScreen` and `🏟️⛺👥📱` on `Dashboard` look amateur next to the Lucide icon set used in the modern screens. They also fail to scale with Dynamic Type and don't match brand colors. | Replace with `lucide-react-native` icons (`Volleyball`, `Trophy`, `MapPin`, `Tent`, etc.) wrapped in `<IconBadge>`. | `screens/games/CreateGameScreen.tsx:27-32`, `screens/home/DashboardScreen.tsx:94-117` |
| C-6 | **User control & freedom** (Nielsen #3) | 🟠 | Filter chips on `DiscoverScreen` and `SquadsScreen` set `onPress={() => undefined}` — they look interactive but do nothing. The selected `activeSport` state on Discover is also never used to filter `DISCOVER_GAMES`. | Either (a) wire them up — at minimum let them filter the in-memory mock list — or (b) hide them until backed by a real query. Don't ship dead UI. | `screens/discover/DiscoverScreen.tsx:69-77`, `screens/teams/SquadsScreen.tsx:64-87` |
| C-7 | **Recognition over recall** (Nielsen #6) | 🟡 | The bottom-tab labels are not visible from the screens themselves; deep-linked stack screens (`GameDetails`, `BookingDetails`) lose the tab context. There's no breadcrumb or "back to Discover" affordance. | Add a back-aware top bar via `Stack.Screen options={{ headerBackTitle: ... }}` that shows the originating tab name. Consider a `<ScreenHeader variant="detail" backLabel="Discover" />` overload. | `navigation/MainNavigator.tsx:122-253` |
| C-8 | **Aesthetic & minimalist design** (Nielsen #8) | 🟡 | The legacy `actionCard` uses 4 columns of emoji + text, with hard `width: '48%'` — overflows on iPhone SE width and looks dated. | Replace with a `<Card>` + `<IconBadge>` grid that uses a 2-column `gap: spacing.lg` flex layout. | `screens/home/DashboardScreen.tsx:88-119` |
| C-9 | **Help & documentation** (Nielsen #10) | 🟢 | First-time users on `ScannerScreen` see only "Point camera at QR code" — no explanation of what they can scan or what happens after. | Add a 1-line `<EmptyState variant="hint">` above the camera viewfinder describing scenarios (booking check-in, game check-in). | `screens/scanner/ScannerScreen.tsx:38-60` |
| C-10 | **Brand consistency** (Brand Guardian) | 🟠 | The wordmark color, weight, and case differ on Login (`Sports Yeti`, plain bold), Discover header (`SportsYeti` in `displayBold` deep blue), and Profile header (`Profile` in `displayBold` deep blue replacing the wordmark). The brand has no consistent lockup. | Decide one wordmark spelling + lockup. Replace ad-hoc `<Text>SportsYeti</Text>` instances with a `<Wordmark size="sm/md/lg">` component. | `screens/auth/LoginScreen.tsx:46`, `ui/ScreenHeader.tsx:54-56` |
| C-11 | **Touch targets (WCAG 2.5.5)** | 🟠 | The `editBtn` (`36×36`) and `viewAllBtn` (no min height) on Profile, the `actionIconWrap` (`48×48` ✅ ok), and several legacy `tab` buttons fall below the iOS HIG 44pt and WCAG 24×24 CSS-px minimum. | Enforce a `<Button minTouchTarget={44}>` rule and audit; bump all icon-only `Pressable` to `≥44×44` via `hitSlop` or padding. | `screens/profile/ProfileTabScreen.tsx:375-383`, `screens/marketplace/MarketplaceScreen.tsx:352-360` |
| C-12 | **Color contrast (WCAG 1.4.3)** | 🟠 | Several token combinations fail AA: `text.muted #94A3B8` on `surface.bg #F6FAFE` = **2.71:1** (fails 4.5:1 normal text). `text.muted` is used on Profile stat labels, day-pill weekday eyebrows. | Either (a) darken `text.muted` to `#6B7785` (4.6:1) or (b) restrict its use to large text (`>18pt regular / >14pt bold`) with a comment in `theme/colors.ts`. | `theme/colors.ts:19`, `screens/profile/ProfileTabScreen.tsx:101` |
| C-13 | **Empty states** (UX best practice) | 🟡 | Empty results just print a sentence. No CTA to "Find a game", "Browse facilities", etc. — dead-end for new users. | Replace with `<EmptyState>` providing illustration/icon, title, body, and primary CTA back into the funnel. | `screens/marketplace/MarketplaceScreen.tsx:299-307`, `screens/messages/MessagesScreen.tsx` (load failure path) |
| C-14 | **Optimistic UI** (Performance perception) | 🟡 | "Join game" awaits the network round-trip with a spinner inside the button, then `Alert.alert('Success')`. The list does not update optimistically — the user has to mentally reconcile. | Optimistically update the local list, show a `<Toast>` on success/failure rollback. | `screens/marketplace/MarketplaceScreen.tsx:88-99` |
| C-15 | **Form validation** (UX best practice) | 🔴 | Login/Register validate on submit only with `Alert.alert`. No inline error, no field-level messaging, no real-time feedback. | Use new `<Input>` with `error` prop driven by `react-hook-form` + `zod` resolver. Validate on blur; show inline messages. | `screens/auth/LoginScreen.tsx:27-38`, `screens/auth/RegisterScreen.tsx` |

## Step 7 — Accessibility QA

### 7.1 New primitives: built-in a11y

Every new primitive ships with the following baseline (see component code for line-by-line):

- **All interactive elements**: `accessibilityRole`, `accessibilityLabel` (or derived from text content), `accessibilityState` for selected/disabled/checked/expanded as relevant, and minimum 44×44 touch target via padding or `hitSlop`.
- **Text uses `<Text>` from `ui/`**: respects user font scaling because variants use `fontSize` not absolute pixel-clamped values.
- **Color contrast**: All token combos used in primitives meet WCAG AA (4.5:1 for normal text, 3:1 for large text and UI components). See §7.3 below for the matrix.
- **Reduced motion**: `Skeleton`, `Toast`, `BottomSheet`, `Modal` honor `AccessibilityInfo.isReduceMotionEnabled()`.
- **Focus order**: Modal/BottomSheet trap focus on iOS via `accessibilityViewIsModal`; the dismiss button is the first focusable element.

### 7.2 Existing primitives: gaps & patches

| Component | Gap | Required patch |
|---|---|---|
| `SearchBar` | `<TextInput>` has no `accessibilityLabel`. Filter button has one ✅ but the text input itself does not. | Add `accessibilityLabel={placeholder}` on the `TextInput`. |
| `Chip` | When `selected`, no `accessibilityState={{ selected: true }}`. | Add `accessibilityState={{ selected }}` when `onPress` is provided. |
| `Button` | `disabled` not communicated to screen reader. | Add `accessibilityState={{ disabled }}`. |
| `Avatar` (image variant) | `Image` has no `accessibilityLabel` — VoiceOver ignores it entirely. | Pass `accessibilityLabel={initials ?? 'Profile photo'}` and `accessibilityRole="image"`. |
| `IconBadge` | Decorative — should be `accessibilityElementsHidden={true}` + `importantForAccessibility="no-hide-descendants"` so the icon doesn't get announced separately from its labeled parent. | Add those two props. |
| `ScreenHeader` | Bell button label says "View notifications" — good. But there's no announcement of the dot. | Append " (1 unread)" to label when `hasNotifications`. |
